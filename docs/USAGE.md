# SDDraft Usage Guide

This guide contains detailed command usage, provider setup, artifact layouts,
and operational notes.

## Commands

- `sddraft generate`
- `sddraft propose-updates`
- `sddraft validate-config`
- `sddraft inspect-diff`
- `sddraft ask`
- `sddraft migrate-index`

Runtime LLM settings can be overridden per command via `--provider`, `--model`,
and `--temperature` (for `generate`, `propose-updates`, and `ask`).
Hierarchy docs are generated by default for `generate` and `propose-updates` and can
be disabled with `--no-hierarchy-docs`.
Engineering graph artifacts are generated by default for `generate` and
`propose-updates` and can be disabled with `--no-graph`.
`ask` uses graph augmentation by default and supports `--no-graph`,
`--graph-depth {1,2}`, and `--graph-top-k <int>`.
Long-running workflows emit `[progress] ...` status lines in the CLI.

## Language Support

- Supported source languages: Python, Java, C++, JavaScript, TypeScript, Go, Rust, and C#.
- Supported extensions: `.py`, `.java`, `.c`, `.cc`, `.cpp`, `.h`, `.hpp`, `.js`, `.mjs`, `.cjs`, `.ts`, `.tsx`, `.go`, `.rs`, `.cs`.
- Parsing uses tree-sitter analyzers in the `repo` layer.
- Unsupported files are analyzed as `unknown` (deterministically, without language-specific parsing) unless excluded by config or `.gitignore`.

## Quick Start

1. Create a virtual environment and install dev dependencies:

```bash
make setup-venv
make install-dev
```

2. Validate configuration:

```bash
sddraft validate-config --project-config examples/project.yaml --csc examples/csc_sddraft.yaml
```

3. Generate an initial SDD:

```bash
sddraft generate --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --repo-root . --provider mock
```

4. Ask grounded questions:

```bash
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "What interfaces are exposed?" --provider mock
```

## Artifact Layout

### Hierarchy artifacts (generated by default)

- `artifacts/<CSC>/hierarchy/manifest.json`
- `artifacts/<CSC>/hierarchy/file_summaries.jsonl`
- `artifacts/<CSC>/hierarchy/directory_summaries.jsonl`
- `artifacts/<CSC>/hierarchy/nodes.jsonl`
- `artifacts/<CSC>/hierarchy/edges.jsonl`
- `artifacts/<CSC>/hierarchy/**` markdown summaries for files/directories

`ask` uses `retrieval/manifest.json` (or the retrieval directory path) and, when present,
expands evidence with `hierarchy/manifest.json`.
If hierarchy artifacts already exist, generation reuses unchanged hierarchy summaries
and only regenerates changed/missing hierarchy nodes.

### Retrieval artifacts

- `artifacts/<CSC>/retrieval/manifest.json`
- `artifacts/<CSC>/retrieval/chunks-*.jsonl`
- `artifacts/<CSC>/retrieval/postings-*.jsonl`
- `artifacts/<CSC>/retrieval/docstats.jsonl`
- `artifacts/<CSC>/run_metrics.json`

### Engineering graph artifacts (generated by default)

- `artifacts/<CSC>/graph/manifest.json`
- `artifacts/<CSC>/graph/nodes.jsonl`
- `artifacts/<CSC>/graph/edges.jsonl`
- `artifacts/<CSC>/graph/symbol_index.json`
- `artifacts/<CSC>/graph/adjacency.json`

`ask` retrieval flow is deterministic and staged:
1. lexical retrieval from `retrieval/`
2. hierarchy expansion (if present)
3. graph anchor extraction + neighborhood expansion (if graph store present)
4. deterministic reranking with transparent score components
5. grounded structured answer generation

If hierarchy or graph artifacts are missing/unreadable, `ask` falls back gracefully
and adds an uncertainty note instead of failing.

## Run On This Repository

Use the built-in SDDraft CSC descriptor to generate documentation for SDDraft itself:
`examples/csc_sddraft.yaml` (`csc_id: SDDRAFT_CORE`).

### Option A: Mock Provider (offline/deterministic)

```bash
sddraft validate-config --project-config examples/project.yaml --csc examples/csc_sddraft.yaml
sddraft generate --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --repo-root . --provider mock
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "What are the main workflows in this project?" --provider mock
sddraft inspect-diff --commit-range HEAD~1..HEAD --repo-root .
sddraft propose-updates --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --existing-sdd artifacts/SDDRAFT_CORE/sdd.md --commit-range HEAD~1..HEAD --repo-root . --provider mock
```

### Option B: Ollama Provider (local model)

```bash
sddraft validate-config --project-config examples/project_ollama.yaml --csc examples/csc_sddraft.yaml
sddraft generate --project-config examples/project_ollama.yaml --csc examples/csc_sddraft.yaml --repo-root . --provider ollama --model qwen2.5:14b-instruct-q4_K_M
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "What are the main workflows in this project?" --provider ollama --model qwen2.5:14b-instruct-q4_K_M
sddraft inspect-diff --commit-range HEAD~1..HEAD --repo-root .
sddraft propose-updates --project-config examples/project_ollama.yaml --csc examples/csc_sddraft.yaml --existing-sdd artifacts/SDDRAFT_CORE/sdd.md --commit-range HEAD~1..HEAD --repo-root . --provider ollama --model qwen2.5:14b-instruct-q4_K_M
```

Generated self-documentation artifacts are written under `artifacts/SDDRAFT_CORE/`.

## Gemini Setup (Optional)

Gemini is supported through the provider abstraction and is optional for local/offline development.

1. Install Gemini support:

```bash
pip install -e .[gemini]
```

2. Set API key:

```bash
export GEMINI_API_KEY="your-key"
```

3. Run with Gemini:

```bash
sddraft generate --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --repo-root . --provider gemini --model gemini-2.5-flash
```

If Gemini dependencies are not installed or `GEMINI_API_KEY` is missing, the CLI exits with a clear `Error:` message.

## Ollama Setup (Local, Optional)

Ollama is supported through the same provider abstraction and uses the local
HTTP API (`/api/chat`) with schema-constrained JSON output.

1. Start Ollama:

```bash
ollama serve
```

2. Pull the recommended local model:

```bash
ollama pull qwen2.5:14b-instruct-q4_K_M
```

3. Run SDDraft commands with Ollama:

```bash
sddraft generate --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --repo-root . --provider ollama --model qwen2.5:14b-instruct-q4_K_M
sddraft propose-updates --project-config examples/project.yaml --csc examples/csc_sddraft.yaml --existing-sdd artifacts/SDDRAFT_CORE/sdd.md --commit-range HEAD~1..HEAD --repo-root . --provider ollama --model qwen2.5:14b-instruct-q4_K_M
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "What interfaces are exposed?" --provider ollama --model qwen2.5:14b-instruct-q4_K_M
```

4. Optional endpoint override (default is `http://127.0.0.1:11434`):

```bash
export OLLAMA_BASE_URL="http://127.0.0.1:11434"
```

If Ollama is unreachable, the CLI exits with `Error: Cannot connect to Ollama ...`.

Example Ollama config is provided in `examples/project_ollama.yaml`.

## Graph-Aware Ask Options

```bash
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "Where is graph expansion implemented?" --provider ollama --model qwen2.5:14b-instruct-q4_K_M --graph-depth 2 --graph-top-k 16
sddraft ask --index-path artifacts/SDDRAFT_CORE/retrieval --question "Which sections are grounded by commit-impact evidence?" --provider mock --no-graph
```

Use `--no-graph` to force lexical/hierarchy-only behavior.

## Vector-Ready Retrieval Seam

Retrieval orchestration uses candidate-source abstractions:
- lexical candidate source (active)
- graph expansion candidate source (active)
- vector candidate source (placeholder interface)

Vector retrieval can be added later without rewriting the `ask` workflow contract.

## Legacy Index Migration

Older artifacts may still have a single-file `retrieval_index.json`.
Current commands require the sharded retrieval store under `artifacts/<CSC>/retrieval/`.

```bash
sddraft migrate-index --index-path artifacts/SDDRAFT_CORE/retrieval_index.json
```

After migration, use `--index-path artifacts/SDDRAFT_CORE/retrieval` with `sddraft ask`.

## Large-Repo Tuning

Use `generation` settings in project config to bound memory and control write patterns:

- `write_batch_size`: flush cadence for streaming JSONL writes.
- `max_in_memory_records`: upper bound for in-process buffered records.
- `index_shard_size`: per-shard chunk target for retrieval store files.

## Quality Gates

- `mypy` type checking
- `pytest` + `pytest-cov` with coverage threshold

Run all checks with:
